### Notes on Route Groups

- `(auth)/` вҖ” parentheses make this a route group; the folder name does not appear in the URL. `/login` renders at `http://localhost:3000/login`, not `/auth/login`.
- `dashboard/` вҖ” no route group wrapper; path segments appear in URL as-is (`/dashboard/employee`, `/dashboard/hr`, `/dashboard/owner`).
- `api/v1/auth/[...nextauth]/route.ts` вҖ” the catch-all route handles NextAuth's internal `/api/v1/auth/callback/credentials`, `/api/v1/auth/session`, `/api/v1/auth/csrf`, etc.

## 3. Authentication Library вҖ” Recommendation and Rationale

### Decision: NextAuth.js v4

**Rejected alternatives and why:**

| Option | Reason Rejected |
|--------|----------------|
| **Auth.js v5 (NextAuth v5)** | In beta / RC as of last stable release; API surface still changing; session adapter integration differs significantly from v4; not suitable for a production system built today |
| **Clerk** | External SaaS; requires internet for auth; no control over user data; cannot enforce HR approval gate at the auth provider level; additional monthly cost |
| **Auth0** | Same as Clerk вҖ” third-party dependency, data leaves the system |
| **Supabase Auth** | Couples auth to Supabase infrastructure; we use PostgreSQL independently via Prisma |
| **Custom JWT** | Requires building CSRF protection, token rotation, secure cookie handling, and session refresh manually вҖ” all of which NextAuth provides for free |

**Why NextAuth v4:**

1. **Battle-tested**: Deployed in production across thousands of Next.js applications; excellent community support and extensive documentation.

2. **Credentials provider**: Supports email + password out of the box. The `authorize()` callback is where our bcrypt comparison and account status check (`status === 'APPROVED'`) live.

3. **JWT strategy**: Using `strategy: "jwt"` avoids the need for a `sessions` table in PostgreSQL. Session data is stored in a signed, HTTP-only cookie on the client. This aligns with our schema (no sessions table defined).

4. **CSRF protection**: Built-in double-submit cookie pattern on sign-in. Our custom API routes (`POST /api/v1/attendance/action`) rely on the session cookie which NextAuth manages securely.

5. **Session callbacks**: The `jwt()` and `session()` callbacks allow injecting custom fields into the token: `id`, `role`, `status`, `fullName`. These are read by middleware using `getToken()` at the Edge without any database call.

6. **Custom pages**: `pages.signIn = "/login"` redirects all auth flows to our custom `/login` page, not NextAuth's built-in page.

7. **App Router compatibility**: `getServerSession(authOptions)` works in Server Components, route handlers, and server actions. Edge middleware uses `getToken()` which reads the same JWT cookie.

**Key configuration decisions:**

| Setting | Value | Reason |
|---------|-------|--------|
| `strategy` | `jwt` | No DB sessions table needed |
| `maxAge` | `28800` (8 hours) | NFR-09 |
| `pages.signIn` | `/login` | Custom login page |
| `providers` | `[CredentialsProvider]` | Email + password only |
| `secret` | `NEXTAUTH_SECRET` env var | Minimum 32 chars; signs JWT |

**TypeScript augmentation** (`types/next-auth.d.ts`):
```typescript
// Extends Session and JWT interfaces to include our custom fields.
// Without this, session.user.role would be a TypeScript error.
declare module "next-auth" {
  interface Session {
    user: {
      id: string
      email: string
      fullName: string
      role: "EMPLOYEE" | "HR" | "OWNER"
      status: "PENDING" | "APPROVED" | "REJECTED" | "DEACTIVATED"
    }
  }
}
declare module "next-auth/jwt" {
  interface JWT {
    id: string
    role: "EMPLOYEE" | "HR" | "OWNER"
    status: "PENDING" | "APPROVED" | "REJECTED" | "DEACTIVATED"
    fullName: string
  }
}
```

## 4. Prisma Setup Plan

### Step-by-Step

**Step 1 вҖ” Install Prisma**
```bash
npm install @prisma/client
npm install -D prisma tsx
```

**Step 2 вҖ” Initialize**
```bash
npx prisma init --datasource-provider postgresql
```
This creates:
- `prisma/schema.prisma` (replace contents with our full schema)
- Updates `.env` with `DATABASE_URL` placeholder

**Step 3 вҖ” Write the schema**  
Replace the generated `prisma/schema.prisma` with the complete schema from `docs/DATABASE_SCHEMA.md`. Includes:
- All four models: `User`, `AttendanceRecord`, `Holiday`, `SystemSetting`
- All four enums: `Role`, `AccountStatus`, `AttendanceStatus`, `AttendanceStep`
- All indexes, cascade rules, and relation names

**Step 4 вҖ” Create the initial migration**
```bash
npx prisma migrate dev --name initial
```
Prisma generates `prisma/migrations/20260624000000_initial/migration.sql`.

**Step 5 вҖ” Add the partial unique index**  
The partial unique index on `holidays(date) WHERE deleted_at IS NULL` cannot be expressed in the Prisma schema. It must be added manually to the generated migration SQL file before applying:

```sql
-- Appended to the end of migration.sql after Prisma's generated statements:

-- Partial unique index: only one active holiday (deleted_at IS NULL) per date.
-- Allows the same date to be re-added after a soft delete.
CREATE UNIQUE INDEX "holidays_date_active_unique"
  ON "holidays" ("date")
  WHERE "deleted_at" IS NULL;
```

After editing, apply the migration:
```bash
npx prisma migrate dev
```
This applies the edited SQL. Prisma records it in `_prisma_migrations` so it is not re-applied.

**Step 6 вҖ” Verify the index was created**
```bash
npx prisma db execute --stdin <<< \
  "SELECT indexname, indexdef FROM pg_indexes WHERE tablename = 'holidays';"
```
Expected output includes `holidays_date_active_unique` with the `WHERE (deleted_at IS NULL)` predicate.

**Step 7 вҖ” Generate the Prisma client**
```bash
npx prisma generate
```
Regenerate after any schema change. Always run this before `npm run dev` if schema was changed.

**Step 8 вҖ” Create the singleton client** (`lib/prisma.ts`)

The singleton prevents multiple PrismaClient instances during Next.js hot reload in development (which would exhaust PostgreSQL connections):

```typescript
import { PrismaClient } from "@prisma/client"

const globalForPrisma = globalThis as unknown as { prisma: PrismaClient }

export const prisma =
  globalForPrisma.prisma ?? new PrismaClient({ log: ["error"] })

if (process.env.NODE_ENV !== "production") {
  globalForPrisma.prisma = prisma
}
```

**Step 9 вҖ” Write the seed script** (`prisma/seed.ts`)

The seed script:
1. Creates the Owner user with bcrypt-hashed password (cost 12), `role = OWNER`, `status = APPROVED`
2. Uses `upsert` so running seed twice does not create duplicate Owner records
3. Inserts 4 `system_settings` rows using `upsert` on the `key` field

Reads credentials from: `SEED_OWNER_EMAIL`, `SEED_OWNER_PASSWORD`, `SEED_OWNER_FULL_NAME` env vars.

**Step 10 вҖ” Register seed in package.json**
```json
"prisma": { "seed": "tsx prisma/seed.ts" }
```

**Step 11 вҖ” Run the seed**
```bash
npx prisma db seed
```

## 5. PostgreSQL Setup Plan

### Recommended: Docker Compose (Local Development)

Docker Compose gives every developer an identical, reproducible database environment without installing PostgreSQL locally.

**`docker-compose.yml`** (at project root):
```yaml
version: "3.9"
services:
  postgres:
    image: postgres:15-alpine
    container_name: attendance_db
    restart: unless-stopped
    environment:
      POSTGRES_USER: attendance_user
      POSTGRES_PASSWORD: attendance_pass
      POSTGRES_DB: attendance_db
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U attendance_user -d attendance_db"]
      interval: 5s
      timeout: 5s
      retries: 5

volumes:
  postgres_data:
```

**Start the database:**
```bash
docker compose up -d           # Start in background
docker compose ps              # Confirm status = healthy
docker compose down            # Stop (data is preserved in volume)
docker compose down -v         # Stop AND delete data (full reset)
```

**Connection string (matches .env):**
```
DATABASE_URL="postgresql://attendance_user:attendance_pass@localhost:5432/attendance_db"
```

### Alternative: Hosted PostgreSQL (No Docker)

### PostgreSQL Version

Must be **PostgreSQL 15 or later** because:
- `gen_random_uuid()` is a built-in function since PostgreSQL 13 (no `uuid-ossp` extension needed)
- Partial indexes (used for holidays) are available since PostgreSQL 8 вҖ” no version concern here
- `pg_isready` healthcheck is available since PostgreSQL 9.3

## 7. shadcn/ui Setup Plan

### Why shadcn/ui

shadcn/ui is not a component library installed as a package вҖ” it copies component source code directly into the project (`components/ui/`). This means:
- Components are **fully customisable**: edit the source freely
- No version lock-in: Tailwind and Radix UI do the heavy lifting
- Built on **Radix UI primitives**: accessible by default (ARIA, keyboard navigation)
- **TypeScript first** with CVA (class-variance-authority) for variant management
- No runtime overhead from unused components: only install what you use

### Initialisation

```bash
npx shadcn-ui@latest init
```

**Interactive prompt choices** (Sprint 1 decisions):

| Prompt | Choice | Reason |
|--------|--------|--------|
| Style | **New York** | Cleaner, more compact aesthetic suited to business dashboards |
| Base color | **Slate** | Neutral, professional вҖ” works well for an HR/attendance system |
| Use CSS variables? | **Yes** | Enables dark mode support in V2 without refactoring |
| Tailwind config path | `tailwind.config.ts` | TypeScript config |
| Global CSS path | `app/globals.css` | App Router location |
| Component alias | `@/components/ui` | Matches folder structure |
| Utility alias | `@/lib/utils` | Where `cn()` lives |
| RSC support | **Yes** | App Router uses React Server Components |

This creates `components.json` at the root and updates `tailwind.config.ts` and `app/globals.css` with shadcn/ui variables.

### Components Installed in Sprint 1

## 8. Role-Based Routing Strategy

Route protection operates at three independent layers. Each layer is a separate line of defence.

### Layer 1 вҖ” Next.js Edge Middleware (`middleware.ts`)

Runs before every request, at the Edge (no Node.js runtime, no Prisma). Reads the signed JWT cookie using `getToken()` from `next-auth/jwt`.

**Route matcher** (configured in middleware `config.matcher`):
```
Matches ALL paths EXCEPT:
  /api/v1/auth/*            вҶҗ NextAuth internal routes (public)
  /_next/static/*           вҶҗ Next.js static assets
  /_next/image/*            вҶҗ Next.js image optimisation
  /favicon.ico
```

**Decision logic**:

```
Incoming request
в”Ӯ
в”ңв”Җв”Җ Token missing (not logged in)?
в”Ӯ     в””в”Җв”Җ Redirect вҶ’ /login
в”Ӯ
в”ңв”Җв”Җ Token present but status вү  APPROVED?
в”Ӯ     в””в”Җв”Җ Redirect вҶ’ /pending
в”Ӯ
в”ңв”Җв”Җ Accessing /login or /register while APPROVED?
в”Ӯ     в””в”Җв”Җ Redirect вҶ’ role-appropriate dashboard (see table below)
в”Ӯ
в”ңв”Җв”Җ EMPLOYEE accessing /dashboard/hr/* or /dashboard/owner/*?
в”Ӯ     в””в”Җв”Җ Redirect вҶ’ /dashboard/employee
в”Ӯ
в”ңв”Җв”Җ HR accessing /dashboard/employee/* or /dashboard/owner/*?
в”Ӯ     в””в”Җв”Җ Redirect вҶ’ /dashboard/hr
в”Ӯ
в”ңв”Җв”Җ OWNER accessing /dashboard/employee/* or /dashboard/hr/*?
в”Ӯ     в””в”Җв”Җ Redirect вҶ’ /dashboard/owner
в”Ӯ
в””в”Җв”Җ All other cases: allow request to proceed
```

**Role вҶ’ default dashboard mapping**:

| Role | status = APPROVED | status вү  APPROVED |
|------|-------------------|------------------|
| `EMPLOYEE` | `/dashboard/employee` | `/pending` |
| `HR` | `/dashboard/hr` | `/pending` |
| `OWNER` | `/dashboard/owner` | n/a (Owner is always APPROVED) |

**Why Edge Middleware**:
- Runs before any rendering вҖ” no wasted server work on unauthorised requests
- `getToken()` reads from the signed cookie without a DB call вҖ” zero latency overhead
- Handles both page routes and API routes in a single place

### Layer 2 вҖ” API Route Guards (`lib/auth-helpers.ts`)

Every custom API route handler (`/api/v1/*`) calls a guard function as its very first line. This is the server-side defence вҖ” if Layer 1 is somehow bypassed, Layer 2 blocks the request.

```typescript
// lib/auth-helpers.ts
export async function requireRole(
  allowedRoles: Array<"EMPLOYEE" | "HR" | "OWNER">
): Promise<Session["user"]>
```

Usage in every API route:
```typescript
// Example: /api/v1/hr/employees/route.ts
export async function GET(req: NextRequest) {
  const user = await requireRole(["HR", "OWNER"])  // throws 401/403 if invalid
  // ... handler logic
}
```

**`requireRole()` logic**:
1. Call `getServerSession(authOptions)`
2. If no session вҶ’ return `NextResponse.json({ error: "Unauthorized", code: "UNAUTHORIZED" }, { status: 401 })`
3. If `session.user.status !== "APPROVED"` вҶ’ return `403`
4. If `session.user.role` not in `allowedRoles` вҶ’ return `403`
5. Otherwise вҶ’ return `session.user` for use in handler

### Layer 3 вҖ” Data-Level Filtering (Query Level)

Prevents IDOR (Insecure Direct Object Reference) attacks where an authenticated user crafts a request to access another user's data.

| Endpoint type | Filter applied |
|---------------|---------------|
| Employee self-service (`/api/v1/attendance/*`) | All queries include `WHERE user_id = session.user.id` |
| HR employee list (`/api/v1/hr/employees`) | Results exclude `role IN ('HR', 'OWNER')` вҖ” HR cannot see other HR accounts |
| HR attendance (`/api/v1/hr/attendance`) | No user-id restriction, but validates requested `userId` exists and is an EMPLOYEE |
| Owner HR accounts (`/api/v1/owner/hr-accounts`) | Results filter `role = 'HR'` only |

### Post-Login Redirect

NextAuth's `redirect` callback and the `callbackUrl` parameter handle where users land after login:

```
After successful login:
  role = EMPLOYEE, status = APPROVED  вҶ’  /dashboard/employee
  role = HR,       status = APPROVED  вҶ’  /dashboard/hr
  role = OWNER                        вҶ’  /dashboard/owner
  status вү  APPROVED                   вҶ’  /pending
```

This is implemented in the NextAuth `redirect` callback in `lib/auth.ts`.
